摘要

大语言模型技术发展飞快，智能代码生成成了软件工程里的一个热门方向。像GitHub
Copilot这类工具，能把自然语言描述转成可运行的代码，确实提高了开发效率。不过，现在的"单次提示生成"方式，遇到模糊或不完整的需求时，生成的代码很可能会出错。所以说，要想让这项技术真正用在生产环境里，关键得解决需求模糊的问题。

本文基于交互式需求澄清的思路，重点研究了模糊需求识别、澄清问题生成和多轮对话交互机制的融合设计与实现。具体做了以下四个方面的工作。

第一，分析模糊需求对代码生成质量的影响，说明交互式需求澄清为什么必要；引入"人在回路"理念，搭建"检测---提问---整合---生成"的人机协同工作流，完成可行性分析和总体设计。

第二，研究大语言模型在需求分析上的能力，设计模糊性检测模块；这个模块用两阶段提示策略，从输入规格、边界条件、异常处理等多个方面，自动找出缺失的关键信息。

第三，研究多轮对话管理技术，设计并实现基于有限状态机的对话管理模块和澄清问题生成模块；通过维护对话状态和结构化需求规约，实现系统主动提问和动态整合用户反馈，逐步把需求描述得完整清晰。

第四，研究基于结构化规约的代码生成技术，设计并实现代码生成与解释模块；在需求澄清之后生成高质量的代码，同时给出自然语言解释；还构建了含典型模糊场景的测试数据集，设计对比实验，从代码通过率等方面评估系统性能。

测试结果表明，这个系统能识别需求中的模糊点，引导用户补充关键信息。在10个模糊需求的测试集上，经过交互式需求澄清后，代码通过率（Pass@1）从直接生成的70.0%提升到100.0%，绝对提高了30个百分点，相对提高了42.9%。这说明交互式需求澄清机制能明显提高生成代码的功能正确性，为AI编程工具解决实际应用中的"语义鸿沟"问题提供了一条可行的路子。

**关键词：**交互式需求澄清；智能代码生成；大语言模型；人在回路；人机协同

**Abstract**

Large language model technology is developing very quickly, and
intelligent code generation has become a hot topic in software
engineering. Tools like GitHub Copilot can turn natural language
descriptions into runnable code, which indeed improves development
efficiency. However, the current \"one-shot prompt generation\" approach
is prone to errors when it encounters vague or incomplete requirements.
Therefore, to truly apply this technology in real production
environments, the key is to solve the problem of requirement ambiguity.

This paper focuses on designing and implementing a fusion of vague
requirement identification, clarification question generation, and
multi‑round dialogue interaction, based on the idea of interactive
requirement clarification. The specific work includes the following four
aspects.

First, the impact of vague requirements on code generation quality is
analyzed, and the necessity of interactive requirement clarification is
explained. The \"human‑in‑the‑loop\" concept is introduced, a
\"detect‑ask‑integrate‑generate\" human‑AI collaborative workflow is
built, and feasibility analysis and overall system design are completed.

Second, the capability of large language models in requirement analysis
is studied, and a vagueness detection module is designed. This module
uses a two‑stage prompting strategy to automatically identify missing
key information from several dimensions, such as input specifications,
boundary conditions, and exception handling.

Third, multi‑round dialogue management techniques are studied, and a
dialogue management module based on a finite‑state machine as well as a
clarification question generation module are designed and implemented.
By maintaining dialogue states and structured requirement
specifications, the system can actively ask questions and dynamically
integrate user feedback, gradually making the requirement description
complete and clear.

Fourth, code generation techniques based on structured specifications
are studied, and a code generation and explanation module is designed
and implemented. After requirement clarification, high‑quality code is
generated together with a natural language explanation. A test dataset
containing typical vague scenarios is also constructed, and comparative
experiments are designed to evaluate system performance from aspects
such as the code pass rate.

Test results show that this system can identify vague points in
requirements and guide users to supplement key information. On a test
set of 10 vague requirements, after interactive requirement
clarification, the code pass rate (Pass@1) increased from 70.0% with
direct generation to 100.0% --- an absolute increase of 30 percentage
points and a relative increase of 42.9%. This shows that the interactive
requirement clarification mechanism can significantly improve the
functional correctness of generated code and provides a feasible way to
solve the \"semantic gap\" problem that AI programming tools face in
real‑world applications.

**Keywords**: interactive requirement clarification; intelligent code
generation; large language models; Human-in-the-Loop; human-machine
collaboration

## 1.1 研究背景及意义

### 1.1.1 研究背景

最近几年，用大语言模型来做智能代码生成这项技术，确实有了不小的突破。拿OpenAI
Codex和GitHub
Copilot来说，它们先在大量的代码数据上做了预先训练，然后就具备了能把自然语言描述直接转成可执行代码的能力；正是这个能力，让人工智能辅助软件开发进入了一个新的阶段^\[1\]^。这些工具用起来，能大大提升写代码的效率，而且在那些比较简单、需求也很明确的任务上，它们表现出的性能让人印象深刻。

不过，现在主流的"单次提示生成"这种方式，一碰到真实世界里那些模糊不清、信息不全或者有歧义的自然语言需求，就会暴露出一个根本上的毛病；其根源在于自然语言提示本身作为编程的一种接口，天生就有模糊性，结构也不规整，还特别依赖上下文，并且它还得在一个非确定性的模型上面去执行^\[2\]^。举个例子，用户如果说"写一个排序函数"，这种需求里既没说清楚边界条件，也没给出输入输出是什么格式，更没提性能上有什么要求，那大语言模型只能根据它训练数据里的统计规律，去做一种"概率性的猜测"；这样就极易生成偏离用户真实意图、包含逻辑错误甚至无法编译的无效代码。有研究表明，开发人员要去理解、修改和调试模型生成的代码，会遇到不小的困难，而这些困难又会严重拖累完成任务的效率和成功率^\[3\]^。因此，解决上述由需求模糊性和模型概率性猜测所导致的问题，还要从需求源头提升生成代码的准确性与可用性，这两方面已成为推动该技术从"演示级"迈向"生产级"应用时必须解决的核心瓶颈。

在这种背景下，"人在回路"的交互式理念就给解决这个问题提供了一条关键的思路。借鉴人类软件开发里需求分析师与开发者通过问答澄清细节的协作过程，交互式需求澄清的目的在于把用户被动等着结果的那种模式，给转变成系统主动引导、用户持续反馈的协同创作闭环^\[4\]^。具体来说，如果能构建出一个系统，它会自动识别出需求里哪些地方是模糊的，能聪明地生成出澄清性质的问题，还能把多轮对话管理起来，那这个系统就有希望在真正生成代码之前，先把人和机器的意图给对准了；这样一来，就能显著提升最终产物的质量。本研究所做的，正是顺着这个技术发展方向，进行实践上的探索。

### 1.1.2 研究意义

在当今软件工程领域的发展形势下，大语言模型已成为推动软件开发范式变革的关键动力；随着模型能力的不断增强、应用场景的持续拓展以及工程实践的逐步深入，智能代码生成技术正在迅速渗透到需求分析、代码编写和软件测试等多个开发环节里，特别是在信息技术快速迭代的今天，软件工程领域也在逐渐向智能化与自动化的方向转型；开发并应用智能代码生成系统，一方面顺应了人工智能赋能软件工程的发展趋势，另一方面也有助于提升软件开发的效率与质量，还能进一步推进软件开发流程的智能化进程，因此对降低开发成本、提升软件可靠性具有重要的实践意义^\[5\]^。

在智能代码生成的应用过程中，开发者要面对模糊、不完整甚至带有歧义的自然语言需求，并会因需求表达不清、意图理解困难以及代码调试成本高等问题而感到困扰。通过对开发者的访谈与调研可以验证，上述问题的出现具有一定的普遍性；再通过对代码生成任务全流程的分析，可以总结出以下三个主要方面的问题：第一方面的问题在于，需求表达上存在明显的模糊性，用户往往无法一次性完整且准确地描述所有的功能细节，这会导致模型难以精准理解其真实意图，进而让生成的代码容易出现逻辑偏差或功能缺失，研究表明开发者在理解、编辑和调试模型生成的代码片段时会面临显著困难，这一困难会严重阻碍任务完成的效率与成功率^\[3\]^；第二方面的问题在于模型理解上，大语言模型对自然语言的解析依赖于训练数据的统计规律，当面对像"写一个排序函数"这种缺乏边界条件或性能约束的模糊需求时，模型只能基于其训练数据中的统计模式做出"概率性猜测"，这样就极易生成偏离用户真实意图的代码^\[2\]^；第三方面的问题在于交互方式上，现有工具缺乏有效的反馈与澄清机制，用户无法在生成过程中引导模型理解其隐含意图，只能被动接受生成结果后再反复调试和修改，由此导致了较高的沟通成本和较低的开发效率，难以发挥出人工智能辅助编程的应有价值。

本次毕业设计正是针对以上问题展开的，一是把人与计算机的交互技术以及自然语言处理技术融入到智能代码生成当中，采用了交互式需求澄清的机制来解决需求上的模糊性问题；二是使用大语言模型来识别需求中缺失的信息并主动提问，以此引导用户完善需求的描述；三是将用户的补充回答动态整合为结构化的需求规约，使得模型能够基于明确的需求生成出高质量的代码，从而解决沟通上的障碍并提升代码的生成质量；通过完成基于交互式需求澄清的智能代码生成系统的设计与实现，这一系统能够解决开发者在智能代码生成过程中遇到的需求模糊问题，还能优化人与计算机之间的协作模式，进而更好地辅助软件开发，并展现出人工智能技术给软件工程带来的积极影响。

## 1.2 国内外研究现状

### 1.2.1 HITL研究现状

在人机协同这个核心方向的研究和实践里，HITL正在经历一个很大的转变。以前大家主要盯着怎么把单一的机器学习任务给优化好，现在不一样了，它开始转向给那些复杂的系统工程提供支持。换句话说，现在的研究已经不是光把人类当成数据标注员或者结果校验员那么简单了，而是想在那种动态的、不太确定的环境里面，去搭建一个既紧密配合又能双向增强的人机混合智能系统^\[6\]^。放到智能代码生成这类专业领域里，这个转变意味着系统不光得能理解人类那些模模糊糊的意图，还得有能力主动去引导意图的澄清过程，把人类掌握的知识吸收进来，并且在持续的协作当中，跟人类一起往前走、一起进化。

具体到研究这个层面，目前的进展能看出好几个维度交叉在一起的特点。第一个方面是关于系统架构和设计原则的：研究的重点已经转到了怎么把人类因素给深度地、系统化地塞进信息物理系统这类复杂工程里面去。这就对软件工程提出了更高的要求------不光要想着技术怎么实现，还得把社会技术那个视角也融进去，同时得充分关注人类在环境感知、跟基础设施的交互、多方协调以及最后拍板决策这些环节里的核心作用。第二个方面是方法论上的：HITL这个理念已经渗透到了机器学习的整个流程里头。比如说，在数据预处理阶段，要借助人类的知识去做标注和清洗；到了模型训练阶段，要拿人类给的反馈来优化；再到决策推理阶段，要让人类去审核和修正那些关键输出。举个例子，在数据融合这个领域里，研究者就通过设计HITL机制，让用户能够在实体匹配、聚类纠错和数据转换等多个环节上提供反馈，这样一来，就能用迭代的方式把整个系统的精度和可靠性慢慢提上去^\[7\]^。第三个方面跟大语言模型的兴起有关：HITL研究开始去探索"人在回路"和"模型在回路"这两者融合起来的范式，说白了就是用大模型做中间件，把人类和复杂系统连接在一起，在需求理解、代码生成、结果校验这些环节上，形成一个高效的人机协同闭环^\[8\]^。

在应用领域这块，HITL已经从传统的老领域扩展到了智能制造、智慧医疗、自动驾驶还有个性化教育这些比较前沿的场景里去了。在这些要么风险高、要么交互性很强的领域里，一个系统能不能跑得顺，很大程度上得靠人类的智能去实时理解那些动态变化的环境和模糊不清的边界，并且及时插手干预。尤其是放到智能代码生成这个方向上，HITL的理念正被拿来构建交互式的编程助手------通过多轮对话把用户的需求主动澄清清楚，然后生成出来的代码，质量和可用性都会有一个明显的提升。

### 1.2.2 交互式需求澄清研究现状

交互式需求澄清作为连接人类模糊意图与机器精确执行的关键桥梁，已成为智能软件工程和人工智能交互领域的研究热点。该领域致力于解决的核心问题是：如何让大语言模型等智能系统超越被动的"答题者"角色，通过主动、多轮的对话式交互，逐步理解、挖掘并最终精确匹配用户的真实需求。

看看现在的研究情况，能发现一个趋势，就是从以前那种只盯着单一技术点去突破的做法，慢慢地转向了去做系统化的、可学习的框架。具体来说，可以分成这么几个方面来讲。

第一个方面是关于需求模糊性怎么识别。研究人员已经开始有系统地建一些分类标准了。比如说，有一个叫AmbiTRUS的框架，它从词汇、句法、语义还有语用这四个语言学的层面出发，去识别用户故事里面那些潜在的模糊性问题，而且它还给出了一套系统的分类方法和质量上的标准^\[9\]^。

第二个方面是关于澄清交互的策略。现在的研究不光关心怎么生成问题，还更看重交互过程里头的信息透明和情境共享。有研究就发现，在人机协作的时候，如果给用户提供一些适度的上下文信息（比如让用户知道系统是怎么理解他的意图的），这样就能明显提升协作的效率和用户的体验；这也正好说明了，透明度和共享情境感知这种东西，在交互式澄清里面是很有价值的^\[10\]^。

第三个方面是随着大语言模型能力的快速提升，研究的前沿正在往构建端到端的、可以训练的交互澄清智能体这个方向转。像基于强化学习的UserRL这类框架，它就尝试着把多轮交互抽象成一个序列决策的问题，让模型自己去学什么时候该提问、该怎么去澄清，这样一来，交互策略就能自己慢慢优化了^\[11\]^。

### 1.2.3 大型语言模型（LLM）研究现状

现在大型语言模型的研究，正处在一个关键的转折点上------从以前主要拼规模大小，转到现在更看重能力能不能往深了延伸。过去这一年里，这个领域最明显的一个变化，就是它不再只靠海量数据去做"概率性的文本生成"了，而是开始往具备系统性"逻辑推理"能力的"推理模型"那个方向去转变。这个转变背后最主要的推动力，就是基于可验证奖励的强化学习技术慢慢成熟了，而且也真正用起来了。具体来说，就是把它放到数学、代码这类能自动验证结果的环境里去训练，这样一来，模型就会被促使着去生成中间的那些推理步骤，慢慢地，它就涌现出了有点像人类那样的思维链能力^\[12\]^。拿OpenAI的o1系列和DeepSeek-R1来说，这两代新模型已经表现出了解决复杂数学证明、代码调试还有战略规划这类任务的能力，这些能力可以说彻底地把人工智能能用在哪儿这个边界给拓宽了。

上面说的是能力上的进步，但与此同时，模型规模越弄越大，带来的计算成本也越来越吓人。为了应对这个问题，一场涉及硬件、算法还有软件栈的"效率革命"也在同步进行着。具体表现在以下措施，比如说，有采用更高效计算格式的新型芯片架构，也有像推测解码、KV缓存优化这样的软件算法；这些全链路的优化，目标就是在提升模型能力的同时，把训练和推理过程中的能耗和延迟都给明显降下来。

再来看模型架构，研究也一直在找一些根本性的创新，想突破现在Transformer框架的效率瓶颈。这方面有一个挺有代表性的探索，是谷歌DeepMind这些机构提出来的混合递归架构。这个架构会给不同的Token动态地分配计算深度，这样一来，推理速度就有了明显的提升，同时内存的占用也能减少一半。这也反映出了学术界和工业界的一个共识：光靠堆参数和堆数据规模这种"缩放定律"，已经慢慢遇到边际效益递减的问题了；以后要想有进步，更多的得靠算法和架构底层的创新。

除了架构创新，怎么让模型变得更"聪明"，适应性更强，也是研究前沿。举个例子，Emory大学有研究表明，通过引入记忆机制和动态分配计算资源，大型语言模型在处理相似任务的时候，可以像人一样"熟能生巧"，最后能把推理成本降低高达56%^\[13\]^。再看麻省理工学院，他们提出了一个自适应语言模型的框架，这个框架让模型自己去生成微调的数据，自己更新指令，从而去适应新的任务；这种做法就展现出了模型自我演化的潜力。

这种逐令牌生成方式能够自然建模代码的层次结构，但也容易在生成长序列时累积误差。本系统的llm_clients.py模块通过工厂模式统一封装了DeepSeek、Doubao及Qwen兼容接口等多种模型后端，允许用户在多个模型间灵活切换，正是对这一多元化模型生态的工程适配。

### 2.1.2 训练策略与解码优化

代码生成模型的性能不仅取决于参数规模，还受益于训练策略的改进。填空中间（Fill-in-the-Middle,
FIM）训练将代码补全任务建模为预测被遮蔽的中间部分，极大提升了IDE中实时代码补全的实用性^\[21\]^。指令微调通过构造自然语言指令-代码响应对，使模型能更准确地理解用户模糊或简略的意图^\[22\]^。在解码层面，核采样通过动态截断低概率词元以在多样性和确定性之间取得平衡^\[24\]^；最小贝叶斯风险解码（MBR）则选取与多个参考候选最相似的输出，以降低生成风险^\[25\]^；自我一致性通过多次采样并投票，在数学推理与代码生成任务上显著提升了准确率^\[26\]^。此外，重排序技术通过训练一个奖励模型对多个生成候选评分，然后选取最高分输出，已在代码生成中被证明有效^\[27\]^。本系统在代码生成时采用temperature=0.2的确定性解码，且通过同时生成实验组（澄清后）和对照组（无澄清）代码并对比测试通过率，在效果上简洁地实现了类似"多候选评估"的验证功能。

### 2.1.3 模糊需求下的代码生成挑战

尽管LLM在定义明确的任务上表现优异，但现实世界中的需求往往是不完整、模糊甚至相互矛盾的。多项实证研究揭示了这一根本性局限。Vaithilingam等人通过用户实验发现，使用AI代码助手的开发者需花费大量时间理解、修改和调试生成代码，其根本原因在于自然语言需求本身的模糊性和不精确性^\[3\]^。Meincke等人从自然语言处理的角度系统分析了需求模糊性对代码生成质量的负面影响，指出提示的模糊性、非结构化与上下文依赖性使模型极易生成偏离用户意图的无效代码^\[28\]^。Liu等人在编程竞赛题上进行了大规模测评，证实未经充分澄清的题目描述会导致不同LLM生成代码的功能正确性出现极大方差，且正确率随描述模糊度的增加而急剧下降^\[29\]^。此外，关于代码安全性的研究进一步表明，模糊需求下生成的代码更容易引入安全缺陷。Pearce等人对Copilot生成代码的安全性进行审计后发现，约40%的生成代码存在CWE-Top-25级别的安全漏洞，且当需求未明确定义安全约束时，模型几乎不会主动考虑安全性^\[30\]^。因此，在代码生成工作流中嵌入需求澄清和轻量级安全扫描具有迫切的现实意义。本系统中AnalyzerAgent的模糊点识别和QuestionAgent的逐维度提问，正是一种在生成前主动消解歧义的工程方案；同时，系统在代码生成后自动检测硬编码凭据、eval()调用等危险模式，是对上述安全性研究的直接工程回应。

## 2.2 交互式需求澄清技术

### 2.2.1 需求工程中的模糊性管理

软件需求中的模糊性是一个经典问题。Kamsties等人系统性地研究了自然语言需求规格中的歧义类型，包括词法歧义、句法歧义、语义歧义和语用歧义，并提出了基于检查清单和评审的方法来发现这些问题^\[31\]^。Femmer等人进一步利用机器学习技术自动分类需求文档中的"需求异味"，包括模糊性、主观性和不完整性，该方法在工业案例研究中达到了较高的检测准确率^\[32\]^。近年来，Amna和Poels提出的AmbiTRUS框架从词汇、句法、语义和语用四个语言学层面构建了识别用户故事中潜在模糊性的系统化方法，在116个用户故事上的实证表明该框架具有高覆盖率和低误报率^\[33\]^。这些基础研究为本系统将需求模糊点维度化提供了直接的理论依据------系统将模糊点分解为"输入规格""输出规格""边界条件"等维度，本质上就是将上述研究中提出的语义与语用层面歧义进行可操作的分解。

在需求获取阶段，基于知识图谱和本体的方法也被用来降低模糊性。Dobson和Sawyer从本体论视角重新审视需求对话，认为结构化地提取"实体-属性-约束"是降低需求模糊性的关键^\[34\]^。Aranda等人则发现利用领域本体指导需求访谈能显著提升需求的完备性和一致性^\[35\]^。本系统的技能管理模块允许用户创建包含维度名称、引导问题与默认答案的澄清模板，正是将领域知识以结构化模板形式注入系统，使得模糊性识别不再从零开始，而是基于用户预定义的"知识扩展"来加速澄清过程。

### 2.2.2 基于对话的交互式需求获取

交互式需求获取在对话式AI中有着丰富先例。查询扩展技术通过同义词、概念图谱或用户交互日志扩充原始查询，以缓解模糊查询的低召回率问题^\[36\]^。在代码领域，Specify-by-Example方法让用户提供具体的输入-输出示例来明确需求，大幅降低了描述负担^\[37\]^。在对话式推荐系统中，Christakopoulou等人提出基于主动学习的对话策略，通过选择信息增益最大的问题来快速收敛用户偏好^\[38\]^。这些思想同样适用于代码需求的澄清：系统应能评估当前不确定性，生成高信息量的问题来高效获取用户意图。

更具体的，在智能代码助手中，Ross等人提出了程序合成场景下的对话式澄清框架^\[39\]^，将用户意图建模为潜在的多个可能程序，通过提出区分性测试用例来逐次缩小假设空间。这种方法具有高度的交互信息效率。在人机协作的信息交换研究中，San
Martin等人分析了不同层次的显式信息交换在人机协作中的角色，发现结构化、分层次的信息交换能显著提升协作效率^\[10\]^。本系统的QuestionAgent为每个模糊维度生成引导性问题，正是采用类似思路，通过多轮问答将模糊需求逐步精化为具体规格。

### 2.2.3 对话式澄清中的问题生成策略

在多轮澄清中，如何生成有效的问题是核心挑战。基于信息增益的提问策略旨在选择能最大程度降低不确定性的问题^\[40\]^。在对话状态跟踪中，槽填充方法通过指定目标槽位来驱动系统向用户提问缺失槽值^\[41\]^。基于LLM的问题生成则可以利用模型的语义理解能力，将模糊点描述转化为具体的引导性问题，并提供选项以降低用户回答的认知负荷。本系统采用后一种方案：QuestionAgent接收每个模糊维度的描述，利用LLM生成包含选项示例的具体问题，然后系统逐维度收集用户答案并填充至clarified_spec字典中。这种按槽位逐一提问的方式确保了澄清过程的系统性，避免了开放式提问可能导致的信息冗余或遗漏。

此外，为了避免用户疲劳，系统还应控制提问的数量与质量。Rastogi等人提出的对话式AI设计建议指出，单次会话的提问数量应有所限制，问题应尽可能简洁并提供预设选项^\[42\]^。本系统的前端模糊点编辑器允许用户在正式澄清前删除或合并维度，主动控制了澄清的粒度与数量，符合这一设计建议。Shakeri
Hossein
Abad等人关于交互式纸质原型在需求获取中的研究也表明，用户在可编辑、可视化的需求模型上参与度更高^\[11\]^，这为本系统的模糊点可视化编辑界面提供了设计依据。

### 2.2.4 结构化需求澄清框架：ChatCoder与ClarifyGPT

在LLM代码生成领域，两个代表性框架为交互式需求澄清提供了系统性的方法论。

ChatCoder^\[43\]^设计了一个两轮对话框架------"释义与扩展"和"深入与回溯"，系统性地引导用户完善需求描述，如图2.1所示。

![](media/image4.png)

图2.1 ChatCoder对话框架结构图

第一轮要求LLM从关键概念、方法目的、输入要求、输出要求、边缘情况、异常与错误六个角度扩展需求，用户审查并修正LLM的理解；第二轮则由LLM提出最关心的问题并生成猜测答案，用户逐一确认或修正，并可在发现理解错误时回溯至之前的问题点。实验表明，ChatCoder将GPT-4在Sanitized-MBPP和HumanEval上的Pass@1分别从66.15%和81.10%提升至76.65%和90.24%，同时证明了人工审查在流程中不可或缺。本系统的交互流程直接借鉴了ChatCoder的"扩展-审查-提问-回溯"思想：用户首先编辑和确认模糊点列表（类似于审查LLM的理解），然后逐维度回答系统生成的问题（类似于深入提问），并可通过回滚按钮回溯修正（类似于回溯功能）。

ClarifyGPT^\[44\]^是由Mu等人提出的通过需求澄清增强代码生成的框架，核心创新在于引入了"代码一致性检查"作为需求模糊性的自动判定依据。ClarifyGPT包含四个主要阶段，如图2.2所示。

![](media/image5.png)

图2.2 ClarifyGPT对话框架结构图

ClarifyGPT方法共分为四个步骤：首先，结合大语言模型生成种子输入，并执行基于类型的变异操作来生成测试输入；其次，通过生成多个不同的代码实现，并在相同测试输入上比较它们的输出是否一致------若输出不一致，则判定需求存在模糊性；第三步，将行为不一致的代码解作为示例，引导大语言模型分析模糊点并生成相应的澄清问题；最后，将用户回答与原始需求整合，生成增强后的最终代码。

该框架的核心思想正是利用多个不同代码实现在相同测试输入上的输出差异来检测需求模糊性，实验表明，ClarifyGPT在MBPP-sanitized上取得了13.87%的相对提升，模糊性检测精度达到88.57%。本系统在生成实验组代码的同时也会生成无澄清的对照组代码，并通过"一键测试"对比两组代码的通过率------这实质上是一种轻量级的"一致性检查"------若澄清后通过率显著高于未澄清组，则证明确实消解了原始歧义，反之则提示可能存在未被妥善处理的模糊点。

### 2.2.5 历史经验学习与案例推理

随着系统的持续使用，历史成功案例成为提升生成质量的重要资源。基于案例的推理（Case-Based
Reasoning,
CBR）通过检索历史中与当前问题最相似的案例，将其解决方案复用或适配于新情境^\[45\]^。在代码生成中，CEFC框架从案例库中检索相似代码片段并在结构化知识指导下改写，显著提高了生成质量^\[46\]^。TraceCoder更进一步，不仅存储成功案例，还记录失败轨迹，通过回滚至历史最佳状态进行重试，增强了生成稳定性^\[47\]^。Pan和Zhao的研究进一步从理论上探讨了过往经验能否加速LLM推理的问题，发现结构化的经验检索机制能够显著提升模型的推理效率和准确率^\[13\]^。本系统的HistoryAgent利用词重叠相似度从hllm_memory.json中检索最相似历史案例，并将其作为few-shot示例注入CodeAgent的生成提示中，这正是CBR思想在代码生成提示工程中的直接应用。同时，RollbackAgent维护了最近10次状态快照，允许用户随时回到之前的状态重新决策，为TraceCoder式的探索-回溯提供了基础机制。

### 2.2.6 交互式需求澄清中的可解释性与透明性

在交互式智能系统中，可解释性对于建立用户信任至关重要。Sun等人通过基于场景的设计方法研究了生成式AI在代码生成中的可解释性问题，发现用户在面对AI生成的代码时，强烈期望系统能够透明地展示其推理过程和决策依据，尤其是在代码未能满足预期时^\[1\]^。Meincke等人的提示工程研究报告进一步指出，提示的复杂性和条件性使得用户难以预测模型行为，因此系统应当通过结构化的交互界面降低这种不确定性^\[2\]^。Wu等人提出的AI链概念则展示了通过将大型语言模型提示链接起来，可以实现透明且可控的人机交互^\[8\]^。这些研究为本系统的交互设计提供了重要指导：模糊点编辑界面的可视化表格、实验组与对照组代码的并排展示、代码解释与安全建议的自动生成，都是对可解释性和透明性原则的具体实现。

## 2.3 自动化测试与代码质量评估

### 2.3.1 基于LLM的自动化测试生成

代码生成后的正确性验证是确保系统输出可用的关键环节。传统测试生成主要依赖基于搜索的软件测试技术，如EvoSuite^\[48\]^和Randoop^\[49\]^，它们通过遗传算法或随机方法生成测试输入以最大化覆盖率。近年来，LLM开始在这一领域崭露头角。TestPilot展示了利用LLM为Java方法自动生成JUnit测试的可行性，并通过编译和运行结果筛选低质量测试^\[50\]^。CodaMosa进一步将传统基于搜索的方法与LLM相融合，在覆盖率停滞时调用LLM生成定向测试，成功突破覆盖率瓶颈^\[51\]^。TitanFuzz首次将LLM应用于深度学习编译器测试，展现了模型在复杂测试构造上的潜力^\[52\]^。

本系统实现的"一键测试"功能利用LLM从需求出发自动生成pytest测试用例，继而在临时沙箱目录中执行测试，并通过解析pytest输出计算通过率。这种"生成---测试---评分"闭环，既借鉴了上述工作的核心理念，又通过超时控制和结果解析实现了工程上的轻量化落地。

### 2.3.2 安全沙箱与代码执行隔离

由于生成的代码可能包含危险操作，测试执行必须进行严格的环境隔离。常用技术包括：基于Linux
namespace的轻量进程隔离、Docker容器化、WebAssembly虚拟机及系统调用过滤等。OpenAI在Codex评测环境中采用了基于namespace的沙箱^\[18\]^。本系统为简化部署，采用Python的tempfile.TemporaryDirectory创建临时工作目录，所有文件读写和代码执行均限于此目录，并设置30秒超时以防止死循环耗尽资源。这虽然不如容器化隔离彻底，但对本系统面向的个人与小团队使用场景已提供了足够的安全保障。

### 2.3.3 代码质量的多维度评估

代码质量不仅取决于功能正确性，还涉及可读性、可维护性、效率、安全性等多个维度。自动化评估方面，传统静态分析工具能检测部分代码异味，但在语义层面力有不逮。LLM则能够从更高级的视角进行评价。Pezzè等人对ChatGPT生成代码的系统评估发现，LLM在一定程度上可识别代码可读性问题和潜在缺陷，但与人类专家评估的一致性仍待提升^\[53\]^。在安全性方面，Pearce等人的审计工作^\[30\]^迫使社区正视LLM生成代码中的安全漏洞风险。

本系统采用"自动评分 +
人工修正"的双阶段评估机制：先用LLM对代码进行0-100分自动评分并给出改进建议，随后用户可调整最终评分，评分结果持久化于CodeRating表中。此策略兼顾了自动评估的效率与人类专家判断的准确性，且所收集的用户评分数据为后续基于偏好的模型对齐（如DPO^\[54\]^）提供了标注数据储备。

### 2.3.4 变异测试与测试充分性

变异测试通过向代码注入微小人工缺陷（变异体）并运行测试套件检查其是否能"杀死"这些缺陷，来评估测试充分性。Andrews等人的大规模实证研究表明，变异测试在评估测试套件质量方面优于传统的行覆盖率指标^\[55\]^。虽然本系统当前版本尚未内置变异测试，但其自动测试生成与通过率统计的设计已为此扩展预留了接口。

## 2.4 多智能体协作与人在回路

### 2.4.1 多智能体分工协作架构

随着LLM能力的增强，多智能体协作已成为解决复杂任务的有效范式。ChatDev将瀑布式软件开发流程模拟为一个包含CEO、CTO、程序员、测试员等角色的聊天室，通过自然语言对话完成从需求分析到代码交付的完整生命周期^\[56\]^。MetaGPT通过将软件工程专家的"标准操作流程"以结构化提示注入各个智能体，实现了需求分析、设计、编码、测试的端到端自动化^\[57\]^。Self-Collaboration则让同一模型扮演分析师、编码者和测试者三种角色进行自我对弈，在代码生成任务上取得优于单一生成的效果^\[58\]^。这些工作共同验证了"分而治之"的多智能体范式的有效性。

Liu等人对基于LLM的软件工程智能体进行了全面综述，系统梳理了当前多智能体在需求分析、代码生成、测试调试、项目管理等场景中的应用模式和关键挑战，指出智能体之间的有效协作和信息共享是提升整体效能的核心因素^\[29\]^之扩展。Huang等人提出的知识引导多智能体协作框架进一步展示了将领域知识注入多个智能体以提升需求工程智能化水平的可行性^\[27\]^之原文。借此，本系统的核心业务逻辑正采用这样的设计：Orchestrator作为中央协调器，负责状态转移和调用各专责智能体；AnalyzerAgent专注于模糊点识别；QuestionAgent专注于生成引导性问题；CodeAgent负责代码生成；HistoryAgent管理历史案例；RollbackAgent处理状态回溯。各Agent底层虽可共享LLM，但通过不同的提示模板和职责边界实现了功能分化，使得每个模块能独立优化。

### 2.4.2 人在回路的设计原则与交互准则

"人在回路"（Human-in-the-Loop,
HITL）理念强调在AI系统决策的关键节点保留人的参与。Amershi等人通过对200余篇人机交互研究的分析，提出了AI系统的18条设计指南，其中包括：系统应展示其决策置信度、允许用户反馈、支持高效校正和覆盖^\[59\]^。Clemmensen等人从社会技术视角出发，构建了HITL系统设计的研究框架，提炼出透明性、可控性和互补性三项核心原则^\[60\]^。在具体的交互策略上，Mozannar等人发现，选择合适的求助时机能显著提升用户对系统的信任感与协作效率^\[61\]^。

在代码生成的具体场景中，Weidmann等人开发的VICToRy调试器将人在回路的思想应用于大规模模型转换，通过可视化调试界面允许用户在模型转换的任意阶段介入和修正，显著提升了模型转换的准确性和用户满意度^\[4\]^。本系统的交互设计全面贯彻了上述准则与原则：模糊点编辑阶段允许用户增删改系统推荐的维度，体现可控性；实验组与对照组代码并排展示并附测试通过率，体现透明性；自动评分与用户评分并存，体现互补性。同时，系统通过状态机严格控制逐维度提问的节奏，避免频繁打断用户工作流，回应了Mozannar等人关于提问时机的发现。

### 2.4.3 反馈驱动的模型对齐与持续优化

HITL的深层价值在于利用人类反馈持续优化模型本身。RLHF（从人类反馈中强化学习）通过训练奖励模型拟合人类偏好，再用强化学习微调LLM，显著提升了输出的有用性和安全性^\[62\]^。Ziegler等人进一步优化了RLHF的训练效率^\[63\]^。直接偏好优化（DPO）则直接从偏好数据中优化策略，无需显式奖励模型^\[54\]^。Wang等人提出的信息论强化微调方法，从信息论的角度为LLM的偏好对齐提供了新的理论框架和优化目标^\[12\]^。本系统虽尚未实现RLHF或DPO，但其收集的用户评分和高质量案例标记已为未来使用偏好学习优化代码生成智能体提供了高质量的标注数据源。

## 2.5 对话状态管理与交互策略

### 2.5.1 任务型对话系统的形式化

任务型对话系统旨在通过多轮人机对话帮助用户完成特定任务。POMDP框架为对话管理提供了数学基础，通过信念跟踪在不确定性下进行最优行动选择^\[64\]^。基于有限状态机和槽填充的方法在工业界更为常见，因其行为可预测且易于工程实现。本系统的Orchestrator本质上是一个槽填充式的对话管理器：clarified_spec字典即为槽值集合，每个模糊维度被视为一个待填充的槽，问题索引依次推进直至所有槽值填充完毕，然后触发代码生成。这种FSM设计保证了对话流程的可控性和可解释性，非常适合于本系统这种槽位数量不大且相对固定的澄清场景。

### 2.5.2 对话状态的持久化与恢复

在多轮对话Web应用中，会话上下文的持久化是关键工程问题。常见方案包括服务端会话存储（如Redis）、客户端存储（localStorage/sessionStorage）以及令牌化对话状态。Fielding在其博士论文中提出的REST架构风格对Web应用的状态管理产生了深远影响，其无状态约束要求每个请求包含完成请求所需的所有信息^\[65\]^。本系统采用前后端协同的持久化策略：前端利用sessionStorage暂存消息和代码，后端在每个状态转移后通过orchestrator_state字段将完整快照持久化至MySQL。这既保证了页面刷新时的快速恢复，又保障了长期可靠的状态存档。

### 2.5.3 交互策略的用户适应性

不同用户对交互频率和澄清深度的偏好存在差异。Schubert等人的研究发现，经验丰富的开发者倾向于少量高信息量的提问，而新手则更接受较多引导性问题^\[66\]^。本系统的技能模板机制和模糊点手工编辑功能允许用户根据自己的经验水平定制澄清粒度，从而在一定程度上适应了用户差异性。系统默认的六维度模糊检查清单则为新手提供了结构化的引导框架，降低了使用门槛。

## 2.6 系统架构与工程实现

### 2.6.1 FastAPI框架与RESTful API

FastAPI是一个现代、高性能的Python
Web框架，专为构建API而设计。它底层基于Starlette（提供异步路由和中间件支持）和Pydantic（提供数据校验与序列化），原生支持Python的async/await异步I/O机制^\[67\]^。相比传统的同步框架（如Flask），FastAPI在处理大量并发I/O密集型请求（如数据库查询、外部API调用）时，能够显著提升资源利用率。本系统后端采用FastAPI构建，主要考虑其两大优势：一是原生异步支持，适合需要频繁调用外部LLM服务的高并发场景；二是自动生成OpenAPI文档，便于前后端联调。通过依赖注入机制，认证授权（get_current_user）和数据库会话（get_db）等横切关注点得到优雅处理。API设计遵循RESTful风格，资源语义清晰，便于前后端分离开发。

### 2.6.2 数据库设计与多用户数据隔离

SQLAlchemy是Python领域最成熟的对象关系映射（ORM）框架。它提供高层次的ORM
API和底层的Core
API，允许开发者用Python类和对象操作数据库，同时自动处理SQL语句生成、连接池管理和事务控制。本系统基于SQLAlchemy
ORM管理MySQL数据库。MySQL作为广泛使用的关系型数据库，具备事务ACID特性、索引优化和稳定的主从复制能力，适合存储结构化的用户信息、评测任务记录等核心业务数据。多用户数据隔离通过在所有查询中根据current_user.id进行行级过滤来实现，这是多租户SaaS应用的标准做法，确保每个用户只能访问自己的数据^\[68\]^。此外，orchestrator_state字段采用JSON格式弹性存储非结构化的工作流状态，既利用MySQL的JSON函数实现部分查询能力，又避免了为每种状态单独建表的复杂性，兼顾了性能与灵活性。

### 2.6.3 身份认证与会话安全

JSON Web Token（JWT）是一种开放标准（RFC
7519），它将用户标识、过期时间等声明编码为一个经过数字签名的紧凑字符串。与传统服务端存储Session的方式不同，JWT认证是无状态的，服务器无需维护会话存储，便于横向扩展。本系统采用JWT无状态认证^\[69\]^，用户登录后获得一个由HMAC-SHA256算法签名的令牌（令牌中不包含敏感信息），前端在后续请求的Authorization头部中携带该令牌。用户密码安全方面，系统使用PBKDF2-SHA256密钥派生算法，进行10万次迭代哈希并加盐后存储^\[70\]^。PBKDF2是一种自适应哈希函数，通过重复计算和盐值随机化，能有效抵抗离线暴力破解和彩虹表攻击。前端通过AdminRoute实现基于角色的路由守卫，仅允许具有相应权限的用户访问管理界面；后端在每个API入口再次校验令牌和权限，形成双重安全保障。

### 2.6.4 前端交互与可视化

React是由Meta维护的前端库，采用函数式组件和Hooks的声明式编程范式，配合虚拟DOM技术实现高效的视图更新。本系统前端基于React函数式组件与Ant
Design组件库构建。Ant
Design是一套企业级UI设计语言和React实现，提供了表格、表单、模态框等开箱即用的组件，可快速搭建风格统一的界面。利用react-router-dom实现单页应用（SPA）路由，前端路由切换不刷新页面，提供接近原生应用的交互体验。代码展示模块采用react-syntax-highlighter，该库支持多种编程语言的语法高亮和多种主题样式，增强了LLM输出代码块的可读性。数据分析页面使用Recharts图表库绘制评分趋势折线图和模型对比柱状图。Recharts基于React和D3，声明式地组合图表组件，响应式布局灵活。这些前端技术选型均为现代Web应用的成熟实践，为本系统提供了流畅且信息密度高的用户界面。

## 2.7 本章小结

本章围绕"基于交互式需求澄清的智能代码生成系统"的设计与实现，从大语言模型代码生成、交互式需求澄清、自动化测试与评估、多智能体协作与人在回路、对话状态管理与交互策略、系统架构与工程实现六个方面系统梳理了相关理论技术。首先阐述了LLM代码生成的能力演进及模糊需求下暴露的局限性，明确了引入澄清机制的必要性。随后深入分析了需求工程中的模糊性管理、对话式需求获取、结构化澄清框架（ChatCoder、ClarifyGPT）、案例推理与回滚、可解释性与透明性等核心技术，为系统的交互流程与Orchestrator设计提供了直接依据。接着论述了自动测试生成、安全执行与代码评估等方法，展示了生成-测试-反馈闭环的工程依据。然后剖析了多智能体分工与HITL交互原则，验证了系统架构的合理性和人本导向。最后从Web框架、数据库隔离、认证安全与前端可视化等角度夯实了技术选型基础。上述技术共同构成了本系统的理论支撑体系，为后续章节的需求分析、系统设计及实验评估提供了坚实的理论与实践参考。

# 第3章 系统需求分析

## 3.1 需求分析

这几年软件行业发展飞快，各种应用系统的规模和复杂度都在不停地往上涨，软件开发也慢慢地从原来那种光顾着"写代码"的方式，转向了更看重"工程化协作"的路子上来。但是，在实际做开发的过程当中，开发人员、项目经理还有最终的用户之间，经常会因为需求没说清楚或者大家理解的不一样，导致反复返工、项目往后拖，严重的甚至整个项目就此失败。尤其是对于编程初学者，还有那些自己一个人做开发的独立开发者来说，怎么把脑子里那个模模糊糊的想法，转化成精确的、能真正写出来跑起来的代码规格，是一道突出难题。

根据平时自己写代码时的一些体会，还有观察周围做开发的朋友们的情况，我发现现在的需求分析和代码生成这两个环节，主要存在这么几个方面的问题。

第一个问题是，需求表述得模模糊糊的，缺少一个系统化的澄清办法。用户提需求的时候，大多只说一个很笼统的描述，比如说"给我做个贪吃蛇游戏"，或者说"写一个合并排序的函数"；至于输入到底是什么类型、边界情况怎么处理、出错了怎么办、性能上有没有什么要求，这些关键细节基本上都不怎么提。以前做项目的时候，就只能靠人一遍一遍地去沟通，这样既花时间又费力气，还特别容易漏掉一些重要的方面。

第二个问题是，沟通的成本太高了，因为开发者和用户之间的认知差别很大。两边的知识背景不一样------用户不太懂技术实现那套东西，开发者对业务场景的理解也有限，结果就是澄清需求的那几轮对话效率很低，一些关键的决策点也容易被忽略掉。再加上没有结构化的引导，经常出现一种情况：你以为你已经说清楚了，但实际上根本没说明白。

第三个问题是，从需求到代码这一步的转换，太依赖个人能力了，最后出来的代码质量也是参差不齐。就算需求已经基本上清楚了，不同的开发者写出来的代码，在健壮性、可读性、安全性还有性能这些方面，差别也会非常大。初学者写出来的代码，往往只能应付正常的输入，一碰到异常情况就直接崩溃了，有时候还会忽略掉一些安全隐患。现在市面上虽然已经有了一些通用的大模型辅助编程工具，但大多数都要求用户自己写出特别精确的提示词，并没有真正解决"先把需求想清楚"这个最核心的前提。

针对上面说的这几个问题，本课题设计并实现了一个基于交互式需求澄清的智能代码生成系统。这个系统的想法是，把需求分析和代码生成这两件事紧密地结合在一起。具体做法是，用户先把最初的需求输进去，然后系统会自动识别出这里面有哪些模糊的地方，接着会用一种结构化问卷的方式，一步一步地引导用户把那些缺失的细节给补充上来，直到形成一个完整的、没有歧义的规格说明；之后，系统会调用大语言模型，根据这个规格说明生成符合要求的代码，还会自动去检测一下代码里有没有潜在的安全风险（比如说SQL注入、硬编码的秘钥之类的），然后给出相应的修改建议。这样一来，不仅代码生成的质量提高了，更重要的是，它把需求分析的门槛给降下来了，让那些不是专业技术人员的人，也能比较高效地把自己的想法转成可以运行的代码。

这个系统对于软件工程的教育、低代码开发的推广，还有降低开发者入门的门槛，都是有一定实际意义的；同时，也算是为基于大模型的需求工程工具，摸索出了一条还算可行的路子。

## 3.3 非功能性需求分析

### 3.3.1 性能需求

1.响应时间：前端页面首屏加载不应超过2秒；需求分析（模糊点识别）和代码生成接口的响应时间应在10秒内（视模型API延迟情况，通常应在40秒左右返回）；普通查询（如历史列表、案例列表）应在1秒内完成。

2.并发处理：系统应支持同时至少50个用户的正常使用，后端采用异步框架FastAPI，可充分利用服务器资源，避免阻塞。

3.资源占用：前端构建产物应尽可能小，利用代码分割和懒加载；后端服务内存占用在空闲时应低于200MB，正常使用时不超过1GB。

### 3.3.2 安全需求

1.身份认证：所有敏感接口必须通过JWT令牌验证，令牌设置合理过期时间（默认30天），并在必要时可主动撤销。

2.密码安全：密码采用PBKDF2-SHA256算法加随机盐哈希存储，禁止明文传输或存储。

3.权限控制：普通用户不能访问管理员专属接口（如用户管理、模型配置管理），后端对所有管理接口强制检查is_admin字段。

4.数据隔离：普通用户只能查看自己的历史会话、评分和案例；通过用户ID进行数据过滤。

5.防注入：所有数据库操作使用ORM参数化查询，避免SQL注入；对大模型输入进行简单校验，但主要依赖API侧的安全措施。

6.敏感信息保护：API
Key等配置存储在环境变量或数据库，前端不暴露真实密钥。

### 3.3.3 可靠性需求

1.状态持久化：会话状态（澄清进度、已回答问题）实时保存至数据库，即使用户关闭浏览器或服务重启，重新登录后仍可恢复之前的对话。

2.异常处理：后端对所有可能的异常进行捕获，返回友好的错误信息，避免敏感信息泄露；对大模型API调用设置重试机制（最多2次），防止临时网络波动导致失败。

3.数据备份：数据库应定期备份，建议配置自动备份策略。

### 3.3.4 可扩展性需求

1.模型扩展：可通过后台管理界面直接添加新的大模型，无需修改代码；代码生成代理支持通过配置切换模型。

2.功能扩展：澄清模板、评分标准等均设计为动态配置，管理员可根据需要灵活调整，无需重新部署。

3.前端组件化：React组件结构清晰，页面与业务逻辑分离，新增功能模块时只需增加新路由和页面组件即可。

## 3.4 约束需求

约束需求描述了系统设计和实现中的限制条件：

CR1：后端必须使用Python语言和FastAPI框架。

CR2：前端必须使用React框架，前后端通过RESTful API通信。

CR3：数据库必须使用MySQL。

CR4：API密钥通过环境变量管理，不得硬编码。

CR5：系统需支持至少5轮以上的连续对话，状态不丢失。

CR6：系统内置轻量化一键测试，只限于函数式代码。

## 3.5本章小结

本章对智能代码生成系统进行了全面的需求分析。首先进行了总体的需求分析。然后采用面向对象方法，建立了用户和管理员的业务用例模型，并进一步细化了系统用例图，通过包含、扩展、泛化关系明确了用例间的联系。通过编写关键用例的详细规约表，明确了每个功能的参与者、前置后置条件、基本流程和备选流程。活动图直观展示了核心业务流程的动态逻辑。最后，从性能、安全、可靠性、易用性、可扩展性以及设计约束六个方面提出了非功能性需求，为后续的架构设计、数据库设计和编码实现提供了明确的约束与目标。需求分析过程为系统的成功开发奠定了坚实基础。

# 第4章 系统概要设计

## 4.1 系统总体架构设计

本系统采用前后端分离的B/S架构模式，整体分为表现层、业务逻辑层、数据持久层和AI服务层四个层次。前端专注于用户交互与结果展示，后端负责核心业务逻辑编排与智能体协作。系统总体架构如图4.1所示。

![](media/image12.png)

图4.1 系统总体架构图

1.表现层（前端Web UI）

前端基于React框架构建单页面应用，使用Ant
Design组件库实现统一的界面风格。该层负责提供需求输入与对话交互界面、代码预览与操作界面、用户认证与权限管理界面，以及与后端API的数据通信。前端通过HTTP代理将API请求转发至后端服务，实现前后端分离部署。

2.业务逻辑层（后端FastAPI服务）

业务逻辑层采用Python语言的FastAPI框架构建，提供RESTful风格的API接口。该层的核心是中央协调器（Orchestrator），它采用基于有限状态机的多智能体协作架构，调度五个专业化智能体协同工作：

（1）需求分析智能体（AnalyzerAgent）：负责检测需求中的模糊点。

（2）问题生成智能体（QuestionAgent）：基于模糊点生成澄清问题。

（3）代码生成智能体（CodeAgent）：基于澄清后的规约生成代码和解释。

（4）历史经验智能体（HistoryAgent）：实现历史经验学习机制（HLLM），记录成功案例并支持相似检索。

（5）回滚管理智能体（RollbackAgent）：实现回滚机制，维护状态栈，处理冲突。

3.AI服务层（LLM Service）

AI服务层通过统一的客户端接口封装对不同大语言模型的调用。系统通过数据库中的模型配置表动态管理多个LLM提供商（如DeepSeek、豆包、通义千问等），支持运行时切换模型。LLM服务作为核心推理引擎，为需求分析、问题生成、代码生成等环节提供语言理解与生成能力。

4.数据持久层（Database）

数据持久层采用MySQL关系型数据库进行数据存储，通过SQLAlchemy
ORM框架实现对象关系映射。主要数据实体包括：用户信息表（users）、会话记录表（sessions）、模型配置表（model_configs）、澄清模板表（clarification_templates）、高质量案例表（high_quality_cases）、代码评分表（code_ratings）和需求规格文档表（spec_documents）。此外，历史经验数据以JSON格式持久化存储在文件系统中。

## 4.3 系统工作流程

本系统围绕\"输入需求→检测模糊点→交互澄清→生成代码\"的核心流程展开，各角色在系统中的操作流程如图4.3所示。

![](media/image14.png)

图4.3 系统工作流程图

1.用户输入初始需求。用户通过前端对话界面，以自然语言形式描述期望实现的功能。系统为该用户创建唯一会话标识，并将需求文本提交至后端。

2.模糊点检测。后端接收需求后，需求分析智能体（AnalyzerAgent）采用\"开放式反思+结构化核对\"的两阶段提示策略对需求进行分析。同时，历史经验智能体（HistoryAgent）基于关键词重叠度检索历史相似成功案例，作为参考上下文。

3.判断需求清晰度。若检测结果显示需求描述充分且不存在模糊点，则直接跳转至步骤8进入代码生成环节。

4.展示模糊点并允许编辑。若检测到模糊点，系统将模糊点列表返回前端。用户可以查看、增加、修改或删除模糊点，确认后进入澄清流程。

5.生成并推送澄清问题。问题生成智能体（QuestionAgent）为每个模糊点生成一个具体的引导性问题，系统按顺序逐个推送给用户。

6.用户回答问题。用户输入答案后，系统首先检测该回答是否与已有规约产生冲突。若冲突，回滚管理智能体（RollbackAgent）自动将对话状态恢复至上一个状态并重新提问；若无冲突，则正常整合答案。

7.判断澄清是否完成。系统检查是否所有模糊点均已澄清。若还有未回答的问题，返回步骤5继续推送；若全部完成，则将对话状态切换为\"已确认\"。

8.代码生成。代码生成智能体（CodeAgent）读取澄清后的完整需求规约，构造包含原始需求与澄清细节的提示词，调用LLM生成澄清前后的代码和解释。若存在相似历史案例代码，则作为参考注入提示词。

9.代码测试。内置轻量化一键测试，可直接测试函数代码通过率。

10.经验记录与结果返回。历史经验智能体将本次成功的需求-规约-代码三元组存入经验库。生成的代码和解释返回前端展示，同时自动保存需求规格文档。

## 4.4 对话管理状态机设计

为了规范化管理多轮对话流程，系统采用有限状态机（FSM）对对话状态进行建模。状态机包含六个状态，其转换关系如图4.4所示。

各状态的定义如下：

WAITING_INPUT（等待输入）：初始状态，系统等待用户输入初始需求。

DETECTING（检测中）：系统已收到需求，正在调用分析智能体进行模糊点检测。

EDITING_AMBIGUITIES（编辑模糊点）：检测完成，用户可查看并编辑模糊点列表。

WAITING_CLARIFICATION（待澄清）：用户已确认模糊点列表，系统正逐个推送问题并等待回答。

CONFIRMED（已确认）：所有模糊点均已澄清，需求规约完备，可进行代码生成。

GENERATING_CODE（生成代码中）：系统正在调用代码生成智能体，状态转换完成后回到WAITING_INPUT。

协调器（Orchestrator）作为状态机的实现载体，维护核心数据结构包括：当前状态（current_state）、原始需求（original_requirement）、结构化规约（clarified_spec）、问题列表（current_questions）、问题索引（current_question_index）和对话历史（conversation_history）。

### 4.5.3 表间关系

用户与澄清模板（clarification_templates）：一对多。每个用户可维护自己的模板库，模板数据按user_id隔离。

用户与技能（skills）：一对多。每个用户可创建多个技能。

用户与测评记录（assessment_records）：一对多。每个用户可多次测评（但业务上限制仅一次）。

用户与高质量案例（high_quality_cases）：一对多（通过marked_by外键）。一个用户可以标记多个案例。

会话记录与代码评分（code_ratings）：一对多。一个会话可以进行多次评分（例如重新评分）。

会话记录与需求规格文档（spec_documents）：一对一。每个活跃会话最多生成一份规格文档。

会话记录与测试结果（test_results）：一对多。一个会话可以多次运行测试（每次重新生成测试代码并保存新结果）。

会话记录与高质量案例（high_quality_cases）：一对多。一个会话可以被多次标记为案例（每次更新覆盖）。

技能（skills）与澄清模板（clarification_templates）：多对多，通过中间表skill_templates关联。

测评题目（assessment_questions）与测评记录（assessment_records）：无直接外键，测评记录中以JSON存储答案映射。

## 5.2 需求模糊性检测模块（分析智能体）

### 5.2.1 模块设计原理

需求模糊性检测是本系统的入口模块，其核心功能是分析用户输入的自然语言需求，自动识别其中缺失或模糊的关键信息，并返回结构化的模糊点列表。该模块的实现借鉴了需求工程中启发式评估方法，采用\"开放式反思+结构化核对\"的两阶段提示策略。

第一阶段：开放式反思。
通过提示词引导LLM不设限制地自由思考，从语法、语义、语用等多层面发现潜在模糊之处。提示词设计为：\"你是一个严谨的软件需求分析师。根据以下需求，生成一个JSON数组，每个元素包含\'dimension\'和\'description\'字段，描述需求中的模糊点或不明确的细节。\"这一阶段利用了LLM的零样本推理能力和上下文学习能力，能够捕获意外类型的模糊点。

第二阶段：结构化核对（内置后备机制）。
考虑到LLM输出存在不确定性，模块内部预定义了通用模糊维度清单作为后备方案。当LLM输出无法解析或格式不符合预期时，系统自动降级使用该预定义清单。预定义清单涵盖了需求分析中常见的六个关键维度，如表5.1所示。

表5.1 预定义模糊维度检查清单

  ------------------------------------------------------------------------
  序号    维度                描述
  ------- ------------------- --------------------------------------------
  1       输入规格            输入数据的类型、格式、范围、有效性？

  2       输出规格            输出的类型、格式、返回方式？

  3       边界条件            输入为空、最大、最小值时的行为？

  4       异常处理            如何处理非法输入或运行时错误？

  5       性能约束            是否有时间或空间复杂度要求？

  6       编程语言/框架       期望使用哪种编程语言或框架？
  ------------------------------------------------------------------------

### 5.2.2 模块实现

需求模糊性检测模块的核心实现在AnalyzerAgent类中。该模块接收用户需求和数据库会话对象，通过调用LLM客户端获取分析结果。为保证输出格式的可解析性，提示词中明确要求以JSON数组格式输出，并设置了temperature=0.3以降低输出的随机性。响应处理中采用正则表达式提取JSON部分，并设置了完整的异常捕获和降级机制。

## 5.3 澄清问题生成模块（问题生成智能体）

### 5.3.1 模块设计原理

澄清问题生成模块基于检测出的模糊点列表，为每个模糊维度生成一个具体的引导性问题。模块为每个模糊点独立构造提示词，调用LLM生成问题，确保生成的问题与模糊点一一对应。temperature=0.5的设置是在问题表达的多样性和稳定性之间取得平衡------既避免过于随机的输出导致问题偏离主题，又保留一定的灵活性以产生自然流畅的提问。

### 5.4.3 冲突检测与回滚机制

回滚机制是本系统保证需求规约一致性的关键设计。当用户在后续澄清过程中修改了之前已确认的答案时，若简单地覆盖旧值，可能导致之前基于旧值生成的后续问题失去意义。因此，系统采用回滚策略处理此类冲突。

冲突检测与回滚的完整流程如图5.6所示。

![](media/image23.png)

图5.6 冲突检测与回滚流程图

回滚管理智能体（RollbackAgent）维护一个最大容量为10的状态栈。每次用户回答前，协调器将当前完整状态（包括对话状态、原始需求、规约快照、问题索引）压入栈中。当检测到某维度已有答案且新答案与旧答案不同时，协调器从栈中弹出上一个完整状态并恢复，然后基于原始需求和新答案重新生成问题列表。

## 5.5 历史经验学习机制（历史经验智能体）

### 5.5.1 机制设计原理

历史经验学习机制（HLLM）是本系统实现\"从经验中学习\"的核心设计。其基本原理是将每次成功完成的需求澄清-代码生成过程作为一个经验案例进行持久化存储，并在后续相似需求出现时检索复用。

HistoryAgent负责管理这些历史经验的存储和检索。经验数据以JSON格式持久化存储在hllm_memory.json文件中，每条记录包含原始需求（requirement）、澄清后的结构化规约（clarified_spec）、最终生成的代码（final_code）和质量评分（quality_score）四个字段。

### 5.5.2 模块实现

HistoryAgent在初始化时自动加载已存储的经验文件，提供record_success和get_similar两个核心方法。

（1）经验记录。当代码生成成功后，协调器调用record_success方法，将本次交互的需求、规约和代码作为一条经验记录追加到内存列表中，并持久化保存到JSON文件。经验记录核心实现如表5.8所示。

（2）自动化测试与对比

系统设计了自动化测试模块，用于对实验组与对照组生成的代码进行统一质量评估。该模块集成在CodeAgent中，通过run_automated_test方法完成从可测试性检查到结果存储的全流程。核心步骤包括：

2.测试代码生成：调用LLM生成pytest测试代码，强制首行为fromsolutionimport\*，并明确禁止生成任何包含文件操作、网络请求、系统调用或危险函数（如eval、exec）的测试代码。

4.结果保存：调用/api/test_results/save接口存储通过率及通过标志，供后续在Analysis.js页面中以柱状图形式展示实验组与对照组的总体通过率对比。

### 6.5.2代码通过率验证（对比实验）

数据集来源：实验数据源自GitHubFeatureRequests数据集^\[71\]^。该研究系统分析了开源软件中自然语言需求（Issue与FeatureRequest）的高模糊性与不完整性，为需求澄清研究提供了真实可靠的语料基础。为确保实验的可复现性，我们从中选取了10个代表性需求构建测试任务集，如表6.14所示。

实验设计：

（1）对照组：将原始需求直接输入DeepSeekAPI（temperature=0.2，无任何交互），记录生成代码。

（2）实验组：使用本系统，由研究者扮演用户，根据系统提出的澄清问题提供准确答案（仅补充真实需求中缺失的信息，不改变需求意图），记录最终生成的代码。

# 第7章总结与展望

## 7.1主要工作

本文针对当前智能代码生成技术在处理模糊需求时存在的局限性，设计并实现了一个基于交互式需求澄清的智能代码生成系统。完成的主要工作及存在的不足如下。

第一，关键技术研究与借鉴。
深入研究了ChatCoder、ClarifyGPT等前沿的交互式需求澄清框架，学习并吸收了结构化需求分析、代码一致性检查等核心思想，为系统设计奠定了理论基础。

第二，系统核心模块的设计与实现。
设计并实现了需求模糊性检测、澄清问题生成、多轮对话管理、代码生成与解释、自动化测试对比、技能管理、用户专业测评等核心模块。其中，模糊检测模块采用"开放式反思+结构化核对"的两阶段提示策略，对话管理模块基于有限状态机实现回滚机制与状态持久化，保证了交互过程的稳定性和逻辑性。

第三，原型系统的开发与集成。
采用前后端分离架构，开发出带有网页对话界面的原型系统。前端提供直观的交互体验，后端集成大模型API及测试执行器，实现了从需求输入、澄清交互、代码生成到自动化测试对比的全流程自动化。

第四，实验验证与效果评估。
构建了包含15个典型模糊场景的测试数据集，设计了严格的对比实验。实验结果表明，经过交互式需求澄清后，系统的代码通过率（Pass@1）从直接生成的60.0%提升至100.0%，验证了系统设计的有效性。

与此同时，本系统仍存在以下两方面的主要不足：

不足一：历史经验学习机制较为初级。
当前的历史经验仅基于关键词重叠进行相似检索，并以文本形式作为参考提示注入上下文，未能实现语义级别的相似匹配，也无法自动复用历史案例中的澄清规约来预填充当前对话，导致经验复用的智能化程度较低。

不足二：交互策略缺乏个性化与自适应性。
系统的澄清提问流程完全固定，对所有用户采用相同的提问顺序和问题表述方式，未考虑用户的历史回答习惯、专业水平差异等因素，导致部分情况下交互效率不高。

## 7.2工作展望

未来可以从以下两个方向进行优化和改进。

第一，增强历史经验学习的深度与智能化，实现语义检索与自动预填充。

引入向量数据库（如Chroma、Milvus）对历史成功案例的澄清规约和代码进行语义嵌入，实现基于语义相似度的精准检索，克服当前关键词匹配的局限性。进一步地，将高相似案例的完整澄清规约自动预填充到当前会话中，用户仅需确认或微调，从而大幅减少重复输入，提升经验复用的智能化水平。同时可设计消融实验，量化高层次大模型（如GPT-4）作为历史经验生成器的独立收益。

第二，实现个性化与自适应的交互策略，提升澄清效率与用户体验。

研究基于用户画像的动态提问策略：根据用户的历史回答速度、答案质量、修改频率等行为特征，实时调整澄清问题的复杂度、选项粒度乃至提问顺序，打破当前固定流程的局限。借鉴UserRL等强化学习框架，将用户满意度作为奖励信号，训练系统学习最优的提问时机和提问方式，为不同专业水平的用户提供差异化的澄清路径，从而实现更高效、更自然的交互体验。
